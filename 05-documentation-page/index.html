A fast, modular web fuzzer written in Rust.

Overview

CrabKit is an open-source, high-performance web fuzzer designed for security researchers, penetration testers, and developers who need to discover hidden endpoints, parameters, subdomains, and vulnerabilities in web applications.

Built entirely in Rust, CrabKit leverages the language's zero-cost abstractions and ownership model to deliver throughput that matches or exceeds tools written in C, while being memory-safe by construction. No segfaults. No data races. Just fast fuzzing.

Key Features

  • Async HTTP engine powered by Tokio and reqwest
  • Wordlist-based and mutation-based fuzzing modes
  • Subdomain, path, parameter, and header fuzzing
  • Custom filter rules by status code, response size, and regex
  • Rate limiting and connection pooling
  • Output in plain text, JSON, and JSONL
  • Plugin system via dynamic traits (experimental)

CrabKit is not a vulnerability scanner. It discovers surface area. What you do with that surface area is your responsibility. Always obtain explicit written permission before fuzzing any target.

Installation

Via Cargo (recommended)

Requires Rust 1.75+. Install Rust via rustup.rs.

cargo install crabkit

From Source

git clone https://github.com/example/crabkit.git
cd crabkit
cargo build --release
./target/release/crabkit --version

Verify Installation

crabkit --version

Quickstart

Fuzz paths on a target using a wordlist:

crabkit fuzz \
  --url https://example.com/FUZZ \
  --wordlist /usr/share/wordlists/dirb/common.txt \
  --threads 50 \
  --filter-status 404

The placeholder FUZZ in the URL marks the injection point. CrabKit replaces it with each entry in your wordlist.

Expected output:

[200] /admin          (size: 4821)
[200] /login          (size: 2103)
[301] /static         → /static/
[403] /config         (size: 312)
[200] /api/v1/health  (size: 47)

Done. 4622 requests in 8.3s (556 req/s)

CLI Reference

Global Flags

Global Options
Flag Short Default Description
--threads -t 40 Number of concurrent async workers
--timeout -T 10s HTTP request timeout
--output -o stdout File path for output
--format -f text Output format: text, json, jsonl
--rate-limit -r none Max requests per second (e.g. 100)
--verbose -v off Print all requests, not just matches

Subcommand: fuzz

fuzz — Path and Parameter Fuzzing
Flag Required Description
--url ✅ Yes Target URL with FUZZ placeholder
--wordlist ✅ Yes Path to wordlist file (one entry per line)
--method ❌ No HTTP method: GET (default), POST, PUT, DELETE
--headers ❌ No Custom headers as Key:Value, repeatable
--filter-status ❌ No Hide responses with these status codes (comma-separated)
--match-size ❌ No Show only responses with exact body size in bytes
--follow-redirects ❌ No Follow 3xx redirects automatically

Subcommand: subdomain

crabkit subdomain \
  --domain example.com \
  --wordlist /path/to/subdomains.txt \
  --resolver 1.1.1.1

Subcommand: vhost

crabkit vhost \
  --url https://10.0.0.1 \
  --wordlist /path/to/vhosts.txt \
  --host-header "FUZZ.example.com"

Modules

CrabKit is organized into internal crates. Each module is independently testable.

crabkit-core
The engine. Manages the async task scheduler, connection pool, and request pipeline. Built on Tokio.
crabkit-http
HTTP client abstraction over reqwest. Handles retries, redirects, TLS configuration, and proxy support.
crabkit-wordlist
Wordlist loader and iterator. Supports plain text files, compressed .gz lists, and stdin piping. Memory-maps large files for efficiency.
crabkit-filter
Response filtering engine. Evaluates status code ranges, body size, regex patterns, and response time thresholds.
crabkit-output
Output serializers for text, json, and jsonl formats. Supports concurrent writes to stdout and file simultaneously.
crabkit-plugin (experimental)
Dynamic plugin system using Rust trait objects loaded at runtime. Allows custom request mutation, post-processing, and reporting extensions.

Configuration File

CrabKit reads from ~/.config/crabkit/config.toml if present. CLI flags always override file settings. 

# ~/.config/crabkit/config.toml

[defaults]
threads     = 50
timeout     = "8s"
format      = "json"
follow_redirects = false

[http]
user_agent  = "CrabKit/0.4.2"
proxy       = "http://127.0.0.1:8080"  # optional, e.g. Burp Suite

[filter]
hide_status = [404, 400]
min_size    = 0       # bytes; 0 = no minimum
max_size    = 0       # bytes; 0 = no maximum

Examples

POST body fuzzing

crabkit fuzz \
  --url https://target.com/api/login \
  --method POST \
  --headers "Content-Type:application/json" \
  --data '{"username":"admin","password":"FUZZ"}' \
  --wordlist passwords.txt \
  --filter-status 401

JSON output piped to jq

crabkit fuzz \
  --url https://target.com/FUZZ \
  --wordlist common.txt \
  --format json \
  | jq '.[] | select(.status == 200) | .path'

Fuzzing with a custom User-Agent and proxy

CRABKIT_PROXY=http://127.0.0.1:8080 \
crabkit fuzz \
  --url https://target.com/FUZZ \
  --wordlist paths.txt \
  --headers "User-Agent:Mozilla/5.0 (compatible; MyScanner/1.0)"

Subdomain enumeration

crabkit subdomain \
  --domain example.com \
  --wordlist subdomains-top1m.txt \
  --threads 100 \
  --output results.json \
  --format json

FAQ

Is CrabKit legal to use?
Only against targets you own or have explicit written permission to test. Unauthorized fuzzing is illegal in most jurisdictions. CrabKit is a tool; legality depends entirely on how and where you use it.
Why Rust? Why not Go or Python?
Rust provides native async performance without a garbage collector, which means consistent throughput even under high concurrency. A CrabKit run at 1000 req/s will not pause for GC cycles. It also means the binary is a single self-contained file with no runtime dependency.
How does CrabKit compare to ffuf or feroxbuster?
ffuf is written in Go and is excellent. feroxbuster is also written in Rust and is the closest spiritual relative to CrabKit. CrabKit differentiates itself with its modular crate architecture and the experimental plugin system.
Can I contribute?
Yes. See CONTRIBUTING.md in the repository. All contributions require passing cargo test, cargo clippy, and cargo fmt --check before review.
The fuzzer is slower than expected. What should I check?
  • Increase --threads (default is 40; try 100–200 for fast targets)
  • Check if --rate-limit is set unintentionally in your config file
  • Use CRABKIT_LOG=debug to inspect connection pool behavior
  • Ensure the target is not throttling responses (look at response times in --verbose)
